# Token Tool Strategy

## Baseline

The default tool stack should keep always-on surface area small and route heavier capabilities through explicit workflows.

| Layer | Default owner | Scope |
|---|---|---|
| Shell output reduction | `rtk` | Claude Code, Codex, OpenCode hook equivalents |
| Duplicate-call prevention | `dedup.py` | Claude Code and Codex hooks |
| Claude session continuity | `context-mode` | Claude Code only |
| Lightweight code navigation | `tilth` | Claude Code default when no graph is built |
| Structural code intelligence | `code-review-graph` | Baseline across Claude Code, Codex, OpenCode, Gemini |
| Persistent agent memory | Native memory first | Codex memories, context-mode session index, explicit token-savior workflow only |
| Review and delegation | `codex-plugin-cc` | Claude Code to Codex peer review |

---

## Key Findings

### `dedup.py` origin

`dedup.py` is not generated by `code-review-graph install`. It is a manual port of the DCP duplicate-call strategy into Claude Code and Codex hooks. DCP remains OpenCode-specific.

### Code navigation overlap

`tilth`, `code-review-graph`, and token-savior can all answer code-navigation questions. Do not expose all three as default routing choices.

Use this order:

| Situation | Tool |
|---|---|
| CRG graph exists and status is healthy | `code-review-graph` |
| Claude Code without a CRG graph | `tilth` |
| Codex/OpenCode without a CRG graph | Native search or `rg` through RTK |
| Past memory or decisions | Native memory first; token-savior only by explicit workflow |

### `code-review-graph` gate

`code-review-graph` requires a per-repo `.code-review-graph/` index. A routing instruction to use CRG first is weak unless session start or the agent checks graph status. If no graph exists, the agent must fall back instead of treating empty graph output as evidence.

### token-savior placement

Do not register token-savior as an always-on MCP server in Claude Code or Codex. Its tool surface is large, overlaps with code navigation, and competes with native Codex memories and context-mode. Keep the package optional for explicit recall workflows only.

### Advisor policy

Advisor is an Anthropic API beta tool, not a documented Claude Code `settings.json` baseline. Do not document `advisorModel` as a supported Claude Code setting unless Claude Code documents it.

For custom API harnesses, use `advisor_20260301` with:

| Concern | Policy |
|---|---|
| Pairing | Advisor model must be at least as capable as executor |
| Timing | Call after light orientation, before committing to an approach, and before final delivery on difficult tasks |
| Cost control | Track conversation-level call count client-side |
| Caching | Enable only for long loops expecting at least three advisor calls |
| Failure mode | Executor continues if advisor is unavailable, rate-limited, or over capacity |

Reference: https://platform.claude.com/docs/en/agents-and-tools/tool-use/advisor-tool

## Target Strategy

### Always-on tools

Keep these active by default:

| Endpoint | Always-on |
|---|---|
| Claude Code | `rtk`, `dedup.py`, `context-mode`, `code-review-graph`, `tilth`, `codex-plugin-cc` |
| Codex | native memories, `rtk` hook, `dedup.py`, `code-review-graph` |
| OpenCode | RTK hook equivalent, DCP if installed, code-review-graph via CLI or skill |
| Gemini | code-review-graph MCP wiring |

### Code navigation

Use CRG for PR review, impact analysis, call graphs, refactor planning, and architecture questions when the graph exists. Use tilth for Claude Code file discovery and reading when no graph exists. Use `rg` through RTK when structural tooling is unavailable or would add unnecessary overhead.

### Memory

Codex uses native memories with `memories = true`, `generate_memories = true`, and `use_memories = true`.

Claude Code uses context-mode for session continuity and compaction recovery.

token-savior is installed only when an explicit workflow needs it. It is not part of the default MCP list.

### Review and delegation

Use `/codex:review` for implementation correctness checks and `/codex:adversarial-review` for design or approach challenges. Treat Claude and Codex as peer reviewers. Deterministic tests remain the final evidence.

## Measurement

Use endpoint-specific measurements before removing or promoting tools.

| Endpoint | Measurement |
|---|---|
| Claude Code | `/context-mode:ctx-stats`, `/context-mode:ctx-doctor`, `rtk gain --history` |
| Codex | hook logs, native memory behavior, `rtk gain --history` |
| OpenCode | DCP stats if the plugin is installed |
| CRG | graph status, query usefulness, review/impact quality on built repos |

Cut rule: after at least five real sessions, remove any always-on tool that has zero useful invocations, duplicates another default tool, or adds more routing confusion than value.

For code navigation comparison, choose one repo with a built graph and run the same three tasks through CRG, tilth, and `rg` through RTK. Compare token cost, answer quality, and failure modes before changing defaults.

## Standardization

`install.sh` is the source of truth for default installation. It should:

| Item | Default |
|---|---|
| CRG | Install and register across supported endpoints |
| token-savior | Do not register as default MCP |
| context-mode | Register in Claude Code |
| tilth | Install for Claude Code |
| RTK | Install and initialize hook support |
| Codex | Keep native memories and CRG baseline |

Verify each machine with:

```bash
claude mcp list
codex mcp list
rtk gain --history
```

Expected Claude Code default MCP list: `context-mode` and `code-review-graph`, with no token-savior unless explicitly enabled for a memory workflow.
